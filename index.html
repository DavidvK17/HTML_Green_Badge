The essential feature of a text input is its ability to accept plain text. When we need it to be multiline, nothing changes semantically — that’s why multiline is just a modifier. At the HTML level, this is represented by different markup for technical reasons, which is also fine because we’re only defining semantics, not the implementation. The textfield tag itself is an element, and label is another element; later, we might need other elements, like a status icon, error message placeholder or auto-suggestion.

Sending form data

client (web browser) sends request to server (Apache, Nginx, IIS, Tomcat) using HTTP protocl.
Server answers using the same protocol.

The action attribute

defines where data gets sent - valid relative or absolute URL

The method attribute

defines how data gets sent - GET or POST

The GET method

"Hey server, I want to get this resource"
-> Browser sends an empty body
-> data sent to server is appended do URL

http://www.foo.com?say=Hi&to=Mom
-> once url adress endeded ?
-> followed by name/value pairs
-> each one separated by &



The POST method

"Hey server, take a look at this data in the body and send me back an appropraite result.
data is appended to the body of HTTP request

2 reasons to use POST:

  1. GET displays name/value pairs in URL, also passwords
  2. large amount of data won't fit in URL because browsers limit URL size
